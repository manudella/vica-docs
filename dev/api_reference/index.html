    API Reference¶

    Base URL: http://localhost:8000

    Frontend proxy: /api → backend (via Vite dev server or Nginx).

    System Endpoints¶

    GET /health¶

    Health check. Returns {"status": "ok"}.

    GET /gpu-status¶

    GPU orchestration status: which service owns the GPU, health of each.

    Response: 

    {
  "gpu_mode": "llm",           // "ocr" | "llm" | "idle"
  "ocr_healthy": false,
  "llm_server_healthy": true,
  "llm_model_loaded": true,
  "ocr_container": "exited",   // Docker container status or null
  "ocr_model": "Qwen/Qwen2.5-VL-3B-Instruct",
  "llm_model": "llama3.1:8b"
}

    POST /gpu/ensure-ocr¶

    Switch GPU to OCR mode (stop LLM, start vLLM). Returns {"status": "ok", "gpu_mode": "ocr"}.

    POST /gpu/ensure-llm¶

    Switch GPU to LLM mode (stop vLLM, load Ollama model). Returns {"status": "ok", "gpu_mode": "llm"}.

    POST /gpu/release¶

    Release GPU from all services. Returns {"status": "ok", "gpu_mode": "idle"}.

    GET /api/settings¶

    Read-only view of current application configuration.

    Response: 

    {
  "llm": {
    "enabled": true,
    "model": "llama3.1:8b",
    "base_url": "http://localhost:11434/v1"
  },
  "ocr": {
    "enabled": true,
    "base_url": "http://localhost:8001"
  },
  "debug": true
}

    GET /llm-status¶

    Check LLM connectivity and model availability.

    Response: 

    {
  "status": "ok",              // "ok" | "model_missing" | "unreachable" | "disabled"
  "model": "llama3.1:8b",
  "available_models": ["llama3.1:8b"],
  "error": null
}

    Candidates¶

    GET /candidates¶

    List all candidates with processing status.

    Response: Array of: 

    {
  "id": 1,
  "cv_filename": "john_doe.pdf",
  "has_anonymized_text": true,
  "has_skills": true,
  "job_ids": [1, 2],
  "extraction_method": "llm_deep",    // "llm_deep" | "llm_simple" | "phrase_match" | null
  "analysis_method": "llm"            // "llm" | "fallback_no_llm" | null
}

    POST /candidates/upload¶

    Upload a CV file. Optionally associate with a job.

    Parameters: file (multipart), job_id (query, optional)

    Response: 

    {
  "id": 1,
  "external_id": "abc123",
  "cv_storage_key": "cvs/abc123_john_doe.pdf",
  "cv_filename": "john_doe.pdf",
  "has_anonymized_text": false,
  "has_skills": false
}

    POST /candidates/{candidate_id}/preprocess¶

    Parse CV, agenderize text, section it. Auto-switches GPU to OCR for PDF/image files.

    Response: {"status": "ok", "anonymized_length": 2450}

    POST /candidates/{candidate_id}/extract-skills¶

    Build binary skill vector with evidence. Auto-switches GPU to LLM. Requires preprocess first.

    Response: 

    {
  "status": "ok",
  "skills_count": 15,
  "extraction": {
    "extraction_method": "llm_deep",
    "model": "llama3.1:8b",
    "skills_found": 15,
    "llm_enabled": true
  }
}

    GET /candidates/{candidate_id}/cv¶

    Serve the original CV file (PDF, image, etc.) for viewing. Returns file with appropriate Content-Type.

    GET /candidates/{candidate_id}¶

    Get candidate details.

    Response: 

    {
  "id": 1,
  "external_id": "abc123",
  "cv_storage_key": "cvs/abc123_john_doe.pdf",
  "cv_filename": "john_doe.pdf",
  "has_anonymized_text": true,
  "has_skills": true
}

    Jobs¶

    GET /jobs¶

    List all jobs (excludes validation-run internal jobs). Sorted by creation date descending.

    Response: Array of: 

    {
  "id": 1,
  "title": "Senior Backend Engineer",
  "created_at": "2026-03-20T10:30:00+00:00",
  "candidate_count": 25,
  "has_skills": true
}

    POST /jobs/parse-document¶

    Parse an uploaded JD document (PDF or image). Auto-switches GPU to OCR.

    Parameters: file (multipart)

    Response: {"raw_text": "Job description text...", "content_type": "application/pdf"}

    POST /jobs/¶

    Create a new job from raw text.

    Request body: 

    {
  "raw_text": "We are looking for a Senior Backend Engineer...",
  "title": "Senior Backend Engineer",
  "external_id": null
}

    Response: 

    {
  "id": 1,
  "external_id": null,
  "title": "Senior Backend Engineer",
  "raw_text": "We are looking for...",
  "created_at": "2026-03-20T10:30:00+00:00"
}

    GET /jobs/extract-skills?job_id={id}¶

    AI-extract ordered skill list from JD text. Auto-switches GPU to LLM.

    Response: 

    {
  "skills": [
    {"name": "Python", "position": 0, "required": true},
    {"name": "AWS", "position": 1, "required": true},
    {"name": "Docker", "position": 2, "required": false}
  ]
}

    PUT /jobs/{job_id}/skills¶

    Save recruiter-validated skill ordering.

    Request body: 

    {
  "skills": [
    {"name": "Python", "required": true},
    {"name": "AWS", "required": true},
    {"name": "Docker", "required": false}
  ]
}

    Response: {"status": "ok", "count": 3}

    GET /jobs/{job_id}/candidates¶

    List candidates associated with this job (same format as GET /candidates).

    POST /jobs/{job_id}/candidates¶

    Associate existing candidates with a job.

    Request body: {"candidate_ids": [1, 2, 3]}

    Response: {"status": "ok", "count": 3}

    DELETE /jobs/{job_id}/candidates/{candidate_id}¶

    Remove a candidate from a job's pool. Response: {"status": "ok"}

    GET /jobs/{job_id}¶

    Get job details (same schema as POST response).

    GET /jobs/{job_id}/candidate-statuses¶

    Get recruiting statuses for all candidates in a job.

    Response: {"1": {"status": "interviewed", "updated_at": "..."}, "5": {"status": "hired", "updated_at": "..."}}

    PUT /jobs/{job_id}/candidates/{candidate_id}/status¶

    Set recruiting status for a candidate.

    Request body: {"status": "interviewed"} — valid values: "interviewed", "hired", "rejected", or null to clear.

    Response: {"status": "ok"}

    Match¶

    POST /match/run¶

    Run SPEA2 ranking only (no semantic analysis).

    Request body: 

    {
  "job_id": 1,
  "archive_size": 10,
  "include_global": false
}

    Response: 

    {
  "match_run_id": 1,
  "archive_size": 10,
  "results": [
    {"candidate_id": 5, "rank": 1, "fitness": 4.250, "r_stat": 0.0},
    {"candidate_id": 12, "rank": 2, "fitness": 3.800, "r_stat": 2.0}
  ],
  "remaining": [
    {"candidate_id": 8, "rank": 11, "fitness": 2.100, "r_stat": 5.0},
    {"candidate_id": 3, "rank": 12, "fitness": 1.500, "r_stat": 8.0}
  ]
}

    • results: Top-K candidates (archive), ranked by fitness. These receive full LLM analysis and reports.
    • remaining: All other candidates not in the top-K, ranked by SPEA2 R statistic (ascending). No reports generated for these.

    POST /match/run-and-analyze¶

    Run SPEA2 + stream LLM analysis progress via SSE (Server-Sent Events).

    Request body: Same as /match/run.

    Response: text/event-stream with events:

    Event Data When
    progress {"phase": "ranking", "current": 0, "total": 1, "message": "..."} Start of ranking
    ranking_done Same as /match/run response SPEA2 complete
    progress {"phase": "gpu", ...} GPU switching
    progress {"phase": "analysis", "current": 0, "total": 5, "message": "Analyzing candidate 1/5..."} Per-candidate
    done {...run_result, "analysis": {"analyzed": 5}} All complete

    POST /match/analyze/{match_run_id}¶

    Run LLM analysis on an existing match run's results. Auto-switches GPU to LLM.

    Response: {"status": "ok", "analyzed": 5}

    GET /match/runs/{job_id}¶

    List all match runs for a job (newest first).

    Response: [{"id": 1, "archive_size": 10, "created_at": "...", "result_count": 10}]

    GET /match/latest/{job_id}¶

    Get the most recent match run with full results (including remaining candidates). Returns null if no runs exist.

    Report¶

    GET /report/cv/{candidate_id}/{job_id}¶

    Generate and return an HTML report for a candidate vs job.

    Response: text/html — full standalone HTML page with skill tables, evidence, interview questions, recommendation.

    Validation¶

    POST /validation¶

    Create a validation run.

    Request body: {"title": "Q1 Hiring Round", "run_type": "historical"} — run_type: "historical" or "supervised".

    Response: {"id": 1, "title": "Q1 Hiring Round", "run_type": "historical", "status": "setup"}

    GET /validation¶

    List all validation runs (newest first).

    Response: Array of: 

    {
  "id": 1, "title": "Q1 Hiring Round", "run_type": "historical",
  "status": "completed", "created_at": "...",
  "candidate_count": 50, "has_skills": true
}

    GET /validation/{vr_id}¶

    Get validation run details including stats, recruiter picks, and match run reference.

    DELETE /validation/{vr_id}¶

    Delete a validation run. Response: {"status": "ok"}

    POST /validation/{vr_id}/set-jd¶

    Set or update the job description for a validation run.

    Request body: {"raw_text": "Job description...", "title": "Senior Engineer"}

    Response: {"status": "ok", "job_id": 5}

    GET /validation/{vr_id}/candidates¶

    List candidates in the validation run with recruiting status and recruiter pick flags.

    POST /validation/{vr_id}/run-match¶

    Run SPEA2 + analysis streamed via SSE (same event format as /match/run-and-analyze).

    Optional body: {"archive_size": 50} — defaults to all candidates.

    POST /validation/{vr_id}/recruiter-picks¶

    Set ground truth for supervised validation.

    Request body: {"candidate_ids": [1, 5, 12]}

    Response: {"status": "ok", "count": 3}

    POST /validation/{vr_id}/update-status¶

    Manually update workflow status.

    Request body: {"status": "candidates_ready"} — valid: setup, candidates_ready, jd_ready, processing, matched, completed.

    POST /validation/{vr_id}/compute-stats¶

    Compute and persist validation statistics (precision, recall, F1, etc.).

    Response: Full stats object (see Validation Metrics for field definitions).

    GET /validation/{vr_id}/stats-report¶

    Generate HTML validation statistics report. Response: text/html.

    GET /validation/aggregate/report¶

    Generate aggregate HTML report across all completed validation runs. Response: text/html.
