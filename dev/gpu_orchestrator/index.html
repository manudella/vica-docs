    GPU Orchestration¶

    Problem¶

    VICA uses two GPU-intensive models: - OCR: Qwen2.5-VL-3B-Instruct via vLLM (reads text from PDF/image files) - LLM: llama3.1:8b via Ollama (extracts skills, analyzes candidates)

    On a single-GPU setup (e.g., RTX 5070), both models cannot be loaded into VRAM simultaneously. The GPU orchestrator manages exclusive access.

    Source: app/services/gpu_orchestrator.py

    Three Modes¶

    stateDiagram-v2
    [*] --> IDLE
    IDLE --> OCR: ensure_ocr()
    IDLE --> LLM: ensure_llm()
    OCR --> LLM: ensure_llm()
    LLM --> OCR: ensure_ocr()
    OCR --> IDLE: release_gpu()
    LLM --> IDLE: release_gpu()

    state OCR {
        [*] --> vLLM_Running
        vLLM_Running: vLLM container active
        vLLM_Running: Ollama model unloaded
    }

    state LLM {
        [*] --> Ollama_Loaded
        Ollama_Loaded: Ollama model in VRAM
        Ollama_Loaded: vLLM container stopped
    }

    state IDLE {
        [*] --> No_GPU
        No_GPU: Neither service using GPU
    }
    class GpuMode(str, Enum):
    OCR = "ocr"    # vLLM container running, Ollama model unloaded
    LLM = "llm"    # Ollama model loaded, vLLM container stopped
    IDLE = "idle"   # Neither service using GPU

    OCR Service (vLLM)¶

    Container name: vica-ocr

    Image: vllm/vllm-openai:latest

    Model: Qwen/Qwen2.5-VL-3B-Instruct

    Configuration: - GPU memory utilization: 85% - Served model name: qwen2.5-vl - Max model length: 16,384 tokens - Image limit: 1 per prompt

    Lifecycle: 1. First run: Backend creates the Docker container via docker run. Downloads model (~6GB). Timeout: 300s. 2. Subsequent runs: Backend restarts the existing stopped container via docker start. Timeout: 120s. 3. Stop: Backend stops the container via docker stop (container persists for fast restart).

    Health check: GET /health or GET /v1/models on the OCR base URL.

    LLM Service (Ollama)¶

    Server: Ollama (native or Docker), always running.

    Model: Configured via LLM_MODEL env var (default: llama3.1:8b).

    Load/unload: Ollama keeps models in VRAM until explicitly unloaded. - Load: POST /api/generate with a small prompt triggers model loading into VRAM - Unload: POST /api/generate with keep_alive: 0 releases VRAM

    Health checks: - Server alive: GET /api/tags returns 200 - Model loaded in VRAM: GET /api/ps shows the configured model

    Switching Flow¶

    ensure_ocr()¶

    sequenceDiagram
    participant API as API Request
    participant GPU as GPU Orchestrator
    participant OLL as Ollama
    participant VLLM as vLLM Container

    API->>GPU: ensure_ocr()
    alt OCR already healthy
        GPU-->>API: return True (fast path)
    else Switch needed
        GPU->>GPU: Acquire asyncio lock
        GPU->>OLL: POST /api/generate (keep_alive=0)
        Note over OLL: Unload model from VRAM
        GPU->>GPU: Wait 2s (VRAM release)
        GPU->>VLLM: docker start / docker run
        loop Poll health (120s / 300s first-time)
            GPU->>VLLM: GET /health
        end
        alt Healthy
            GPU->>GPU: Set mode = OCR
            GPU-->>API: return True
        else Failed
            GPU->>VLLM: docker stop
            GPU-->>API: return False
        end
    end

    ensure_llm()¶

    sequenceDiagram
    participant API as API Request
    participant GPU as GPU Orchestrator
    participant VLLM as vLLM Container
    participant OLL as Ollama

    API->>GPU: ensure_llm()
    alt Ollama model already loaded
        GPU-->>API: return True (fast path)
    else Switch needed
        GPU->>GPU: Acquire asyncio lock
        GPU->>VLLM: docker stop
        GPU->>GPU: Wait 3s (VRAM release)
        GPU->>OLL: Wait for server (30s)
        GPU->>OLL: POST /api/generate (preload)
        Note over OLL: Load model into VRAM (120s)
        alt Success
            GPU->>GPU: Set mode = LLM
            GPU-->>API: return True
        else Failed
            GPU-->>API: return False
        end
    end

    release_gpu()¶

    sequenceDiagram
    participant API as API Request
    participant GPU as GPU Orchestrator
    participant OLL as Ollama
    participant VLLM as vLLM Container

    API->>GPU: release_gpu()
    GPU->>GPU: Acquire asyncio lock
    GPU->>OLL: Unload model
    GPU->>VLLM: docker stop
    GPU->>GPU: Set mode = IDLE
    GPU-->>API: done

    Thread Safety¶

    All switching operations go through _gpu_lock = asyncio.Lock(). This serializes concurrent API requests that need different GPU modes — the first request switches, subsequent requests wait and find the GPU already in the right mode.

    Auto-Triggering¶

    API endpoints automatically call the orchestrator before GPU-dependent operations:

    Endpoint Triggers GPU Mode
    POST /candidates/{id}/preprocess PDF/image files need OCR OCR
    POST /candidates/{id}/extract-skills LLM skill extraction LLM
    POST /jobs/parse-document PDF/image JD parsing OCR
    GET /jobs/extract-skills LLM JD skill extraction LLM
    POST /match/run-and-analyze LLM semantic analysis LLM
    POST /match/analyze/{id} LLM semantic analysis LLM
    POST /validation/{id}/run-match LLM semantic analysis LLM

    If the GPU is already in the right mode, the fast path returns immediately (no switch).

    Manual Control¶

    API endpoints for direct GPU management:

    Endpoint Action
    GET /gpu-status Return current mode + health of each service
    POST /gpu/ensure-ocr Force switch to OCR mode
    POST /gpu/ensure-llm Force switch to LLM mode
    POST /gpu/release Release GPU from all services

    Docker Socket Requirement¶

    The backend needs access to the Docker socket to manage the vLLM container:

    # docker-compose.yml
volumes:
  - /var/run/docker.sock:/var/run/docker.sock

    Without this, the backend cannot start/stop the OCR container and will log errors. OCR-dependent operations will fall back to pdfplumber (text-only PDF parsing, no image support).

    Timeouts¶

    Operation Timeout Reason
    vLLM first start 300s Downloads model (~6GB)
    vLLM restart 120s Model already cached
    Ollama server reachable 30s Wait for server startup
    Ollama model preload 120s Load model into VRAM
    Docker stop 30s Graceful container shutdown
    Health check poll 5s per check Individual HTTP request
    Health poll interval 3s (OCR), 2s (Ollama) Between poll attempts
