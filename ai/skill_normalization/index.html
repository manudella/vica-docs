    Skill Normalization Pipeline¶

    How raw skill mentions from CVs and job descriptions become canonical skills, how the ESCO taxonomy works, and how hierarchy enables partial matching.

    Overview¶

    When VICA extracts a skill name (e.g., "Amazon Web Services" from a CV or "Cloud deployment workflows" from a JD), it needs to map it to a standard form so that CV skills and JD skills can be compared. This happens in three stages:

    graph LR
    RAW["Raw Skill Mention"] --> EX{"1. Exact Match?"}
    EX -->|"Yes"| ESCO1["ESCO Canonical<br/>source: esco"]
    EX -->|"No"| FZ{"2. Fuzzy Match<br/>≥ 85%?"}
    FZ -->|"Yes"| ESCO2["ESCO Canonical<br/>source: esco"]
    FZ -->|"No"| OV["Keep as-is<br/>source: other"]

    style ESCO1 fill:#e8f5e9,stroke:#4caf50
    style ESCO2 fill:#e8f5e9,stroke:#4caf50
    style OV fill:#fff3e0,stroke:#ef6c00

    ESCO Taxonomy¶

    Source file: data/taxonomy/esco_skills.json

    Structure: A JSON array of skill objects:

    {
  "id": "skill_001",
  "preferred_label": "Python",
  "alt_labels": ["Python programming", "Python 3"],
  "parent": "skill_000"
}
    Field Purpose
    id Unique identifier
    preferred_label The canonical name (used everywhere in VICA)
    alt_labels Alternative names that resolve to the same canonical
    parent ID of the parent skill (enables hierarchy)

    TaxonomyResolver¶

    Source: app/services/taxonomy.py:26

    Index Building¶

    On initialization, builds three indexes:

    1. Label-to-preferred map: every preferred_label and alt_label (lowercased) → preferred_label
    2. Phrase list: all labels for fuzzy matching
    3. Hierarchy maps: _parent_of (child → parent name) and _children_of (parent → set of children)

    Resolution: resolve(raw_skill) → (canonical_name, source)¶

    1. Exact match:  raw_skill.lower() in label_to_preferred?
                 → Yes: return (preferred_label, "esco")

2. Fuzzy match:  rapidfuzz.process.extractOne(raw_skill, phrases, scorer=ratio)
                 → Score ≥ 85: return (preferred_label, "esco")

3. Open vocab:   → return (raw_skill, "other")

    Fuzzy threshold: 85 (configurable via fuzz_threshold parameter). Uses rapidfuzz.fuzz.ratio scorer.

    Example resolutions:

    Raw input Resolution Source
    "Python" "Python" esco (exact)
    "python programming" "Python" esco (exact via alt_label)
    "Pythoon" "Python" esco (fuzzy, score ~91)
    "FastAPI" "FastAPI" other (not in taxonomy)

    Hierarchical Matching¶

    Source: app/services/taxonomy.py:93 — get_related_skills(canonical_name)

    Returns the skill itself + its parent + its children. This enables partial matching in SPEA2:

    graph TD
    CLOUD["Cloud (parent)"] --> AWS["AWS (child)"]
    CLOUD --> AZURE["Azure (child)"]
    CLOUD --> GCP["GCP (child)"]

    style CLOUD fill:#e3f2fd,stroke:#1565c0
    style AWS fill:#fff3e0,stroke:#ef6c00
    style AZURE fill:#fff3e0,stroke:#ef6c00
    style GCP fill:#fff3e0,stroke:#ef6c00

    Matching behavior:

    Job requires Candidate has Match? Why
    Cloud AWS Yes AWS is a child of Cloud
    AWS Cloud Yes Cloud is the parent of AWS
    AWS Azure No Siblings are NOT matched
    AWS AWS Yes Exact match

    Why no sibling matching: AWS and Azure are different technologies. A candidate knowing AWS doesn't mean they know Azure. The parent-child relationship captures "specific implies general" and "general covers specific" — but not "specific implies other specific."

    This matching is applied in app/api/match.py:64 when building the SPEA2 matrix:

    for s, _ in job_skills_list:
    related_names = resolver.get_related_skills(s.name)
    related_ids = {s.id}
    for rname in related_names:
        rs = store.get_skill_by_normalized(rname.lower())
        if rs:
            related_ids.add(rs.id)
    job_skill_related_ids[idx] = related_ids

    JD Skill Normalization¶

    Source: app/services/jd_extraction.py:198 — normalize_jd_skills()

    After the LLM extracts skills from a job description, three post-processing steps run:

    1. Compound Decomposition¶

    _decompose_compound_skill(name) at line 138:

    Splits multi-skill phrases using connectors ("or", "and", "/"):

    Input Output
    "JavaScript or TypeScript" ["JavaScript", "TypeScript"]
    "Python and Django" ["Python", "Django"]
    "Async processing or task queues" ["Asynchronous Processing", "Task Queues"]
    "Automated testing for backend and frontend" ["Automated Testing"] (strips prepositional clause first)

    Only splits if resulting parts are ≤ 4 words each (avoids splitting sentences).

    2. Qualifier Stripping¶

    _strip_qualifiers(name) at line 159:

    Removes noise prefixes and suffixes that prevent taxonomy matching:

    Input Output
    "CI/CD pipelines" "CI/CD"
    "Cloud deployment workflows" "Cloud"
    "Modern JavaScript" "JavaScript"
    "Advanced Python programming" "Python"

    Iterates up to 5 times until stable (handles "Cloud deployment workflows" → "Cloud deployment" → "Cloud").

    Stripped suffixes: pipelines, workflows, services, tools, frameworks, platforms, design, development, implementation, management, systems, testing, processing, orchestration, databases, etc.

    Stripped prefixes: Modern, Advanced, Basic, Core, Enterprise, Production, Strong, Solid, Deep, Hands-on, Expert, etc.

    3. Taxonomy Resolution + Deduplication¶

    For each skill: 1. Try taxonomy resolution (exact → fuzzy) 2. If matched as "esco": use canonical name 3. If "other" (open vocabulary): try stripping qualifiers and re-resolving 4. If still "other": keep as-is 5. Skip duplicates (case-insensitive, preserving order)

    SkillNormalizer Wrapper¶

    Source: app/services/skill_library.py

    Thin wrapper around TaxonomyResolver. Used throughout the codebase for backward compatibility. The normalize(name) method calls resolver.resolve(name) and returns the canonical name.

    Extending the Taxonomy¶

    To add new skills to data/taxonomy/esco_skills.json:

    1. Add a new object to the JSON array: 

      {
  "id": "custom_001",
  "preferred_label": "LangChain",
  "alt_labels": ["Langchain", "lang chain"],
  "parent": null
}

    2. To make it a child of an existing skill, set parent to the parent's id: 

      {
  "id": "custom_002",
  "preferred_label": "Amazon Bedrock",
  "alt_labels": ["Bedrock"],
  "parent": "skill_aws_id"
}

    3. Restart the backend (taxonomy is loaded at startup)

    Requirements: - id must be unique - preferred_label is the canonical name used everywhere - alt_labels are optional but improve fuzzy matching - parent must reference an existing id or be null
