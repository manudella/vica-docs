    Prompt Templates Reference¶

    All LLM calls in VICA go through call_local_llm() in app/services/llm_client.py. Every call uses: - API: OpenAI-compatible (/v1/chat/completions) - Temperature: 0 (deterministic) - Provider: Local only (Ollama, vLLM, LM Studio)

    1. CV Deep Skill Extraction¶

    Source: app/services/skill_extraction.py:27 — DEEP_SKILL_EXTRACT_PROMPT

    Purpose: Extract all professional/technical skills from a CV with structured evidence.

    System message: The prompt itself (no separate system message).

    User message: CV text (truncated to 32,000 chars).

    Token limit: 16,384 output tokens. Timeout: 300s.

    Prompt text:

    You are an expert technical recruiter analyzing a CV. Extract ALL professional and technical skills you can identify.

For EACH skill, provide:
- "name": the canonical skill name (use standard industry names: "AWS" not "Amazon Web Services",
  "CI/CD" not "CI/CD pipelines", "JavaScript" not "Modern JavaScript").
  One skill per entry — split compound skills like "Python/Django" into separate entries
- "proficiency": one of "expert", "proficient", "familiar", or "mentioned"
  - expert: multiple years of hands-on use, led projects, deep knowledge demonstrated
  - proficient: clear practical experience, used in real projects
  - familiar: some exposure, coursework, or minor usage
  - mentioned: listed but no supporting evidence in the CV
- "evidence": array of ALL direct quotes from the CV that support this skill. CITATION RULES:
  - Quote FULL SENTENCES or full bullet points from the CV, not fragments
  - Include the surrounding context (company name, project, dates) when present in the same sentence or bullet
  - If a skill is mentioned across multiple sections (e.g. work experience AND projects),
    include quotes from EACH section
  - Copy the text EXACTLY as it appears in the CV — do not paraphrase or summarize
  - Each quote should be long enough to stand alone as evidence (typically 1-3 complete sentences)
- "context": brief description of WHERE/HOW the skill was used
  (e.g. "used in backend development at CompanyX for 2 years")
- "justification": one of "strong", "moderate", "weak", "unsubstantiated"
  - strong: multiple concrete examples, projects, or years of experience
  - moderate: at least one concrete example or project
  - weak: mentioned in skills list or implied but limited evidence
  - unsubstantiated: claimed but contradicted or no evidence at all

Return ONLY a JSON object:
{"skills": [{"name": "...", "proficiency": "...", "evidence": ["..."],
  "context": "...", "justification": "..."}, ...]}

Be thorough — extract every skill you can find, including soft skills if clearly evidenced.
Do NOT invent skills that are not in the CV.

    Expected output:

    {
  "skills": [
    {
      "name": "Python",
      "proficiency": "expert",
      "evidence": ["Led migration of legacy Java services to Python 3.11 microservices at ACME Corp (2021-2023)"],
      "context": "Primary backend language at ACME Corp for 3+ years",
      "justification": "strong"
    }
  ]
}

    Output field meanings:

    Field Type Values Purpose
    name string Canonical skill name Used for taxonomy matching and binary vector
    proficiency string expert / proficient / familiar / mentioned Skill depth level (for reports only, not SPEA2)
    evidence string[] Exact CV quotes Shown in candidate reports as citations
    context string Where/how used Brief description for report
    justification string strong / moderate / weak / unsubstantiated Evidence quality (for reports only, not SPEA2)

    Fallback: When LLM is unavailable, falls through to simple extraction (below), then phrase matching.

    2. CV Simple Skill Extraction (Legacy)¶

    Source: app/services/skill_extraction.py:18 — SKILL_EXTRACT_PROMPT

    Purpose: Extract skill names only (no evidence/proficiency). Used as fallback when deep extraction fails.

    Prompt text:

    You extract a list of professional/technical skills from a CV text.
Return only a JSON object with a single key 'skills' and value being an array of skill names (strings).
No explanation, only valid JSON.

    Expected output: {"skills": ["Python", "AWS", "Docker", ...]}

    Token limit: Default. Timeout: Default.

    3. JD Skill Extraction¶

    Source: app/services/jd_extraction.py:26 — JD_SKILLS_PROMPT

    Purpose: Extract ordered skills from a job description, classifying each as required or preferred.

    User message: JD text (truncated to 16,000 chars).

    Prompt text:

    You extract professional/technical skills from a job description and classify each as
"required" or "preferred" based on the JD wording. Skills from 'Required qualifications',
'Must have', or core responsibilities are "required". Skills from 'Preferred', 'Nice to have',
'Bonus', or 'Plus' sections are "preferred". Order by importance (most important first).

CRITICAL RULES for skill names:
1. DECOMPOSE compound phrases into separate atomic skills.
   Example: "Modern JavaScript or TypeScript" → two entries: "JavaScript" and "TypeScript".
   Example: "Automated testing for backend and frontend code" → "Automated Testing".
   Example: "Asynchronous processing or task queues" → "Asynchronous Processing" and "Task Queues".
2. Use SHORT CANONICAL names — the standard industry name for the technology or skill.
   Do NOT include adjectives, qualifiers, or workflow descriptions.
   Example: "Cloud deployment workflows" → "Cloud".
   Example: "CI/CD pipelines" → "CI/CD".
   Example: "REST API design" → "REST APIs".
   Example: "Container orchestration" → "Kubernetes".
   Example: "NoSQL databases" → "NoSQL".
   Example: "Relational databases" → "SQL".
3. If a JD phrase maps to a well-known technology, use that name:
   "AWS services" → "AWS", "Google Cloud" → "GCP", "Microsoft Azure" → "Azure".
Return ONLY a JSON object: {"skills": [{"name": "...", "required": true/false}, ...]}.
No explanation, only valid JSON.

    Expected output:

    {"skills": [{"name": "Python", "required": true}, {"name": "Docker", "required": false}]}

    Fallback: When LLM unavailable, uses _extract_jd_skills_regex() — phrase matching against the skill library.

    Post-processing: After LLM extraction, normalize_jd_skills() applies: 1. Compound decomposition (_decompose_compound_skill): splits "X or Y" / "X and Y" into separate entries 2. Taxonomy resolution: maps to ESCO canonical names when possible 3. Qualifier stripping (_strip_qualifiers): removes noise suffixes/prefixes ("CI/CD pipelines" → "CI/CD") 4. Deduplication preserving order

    4. JD Generation (Testing Only)¶

    Source: app/services/jd_extraction.py:14 — GENERATE_JD_SYSTEM + GENERATE_JD_USER

    Purpose: Generate synthetic job descriptions for testing.

    System message:

    You are an expert HR writer. You produce complete, realistic tech job descriptions
suitable for real job postings. Output only the job description text, no preamble or meta-commentary.

    User message:

    Generate one complete tech job description (e.g. Senior Software Engineer, Data Engineer,
DevOps Engineer, or similar). Include: a clear job title at the top; a short 'About the role' paragraph;
'Responsibilities' (bullet list); 'Required qualifications' (bullet list with concrete technologies and years);
'Preferred qualifications' and 'Nice to have' (optional). Use realistic wording and common tech stack names
(e.g. Python, React, AWS, Kubernetes, PostgreSQL). Keep it professional and 300–600 words.

    Token limit: 2,048. Timeout: 120s.

    5. Candidate Analysis (Post-Match)¶

    Source: app/services/skill_matching.py:22 — CANDIDATE_ANALYSIS_PROMPT

    Purpose: Deep semantic analysis of a candidate against a job. Called once per candidate in the top-K after SPEA2 ranking.

    System message: "You are an expert technical recruiter. Analyze candidates against job requirements. Return only valid JSON."

    User message: The prompt below with variables filled in: - {required_skills}: comma-separated required skill names - {preferred_skills}: comma-separated preferred skill names - {candidate_skill_profile}: formatted deep skills with proficiency, evidence quotes, context - {cv_text}: anonymized CV text (truncated to 24,000 chars)

    Token limit: 16,384. Timeout: 300s.

    Prompt text:

    You are an expert technical recruiter performing a deep analysis of a candidate
against a job description.

## Job Skills
Required skills (MUST have): {required_skills}
Preferred skills (NICE to have): {preferred_skills}

## Candidate's Skill Profile
{candidate_skill_profile}

## Candidate's CV Text
{cv_text}

Analyze this candidate thoroughly and return a JSON object with these sections:

### 1. skill_assessments
For EACH job skill (required AND preferred), assess the candidate:
- "skill": the job skill name
- "required": true/false
- "match_level": one of "strong_match", "partial_match", "transferable", "no_match"
- "evidence_summary": 1-2 sentence summary of relevant CV evidence
- "cv_quotes": array of ALL direct quotes from the CV supporting this skill (CITATION RULES apply)
- "related_candidate_skills": any candidate skills related to this job skill
- "confidence": "high", "medium", or "low"

### 2. transferable_skills
Skills the candidate has that are NOT in the job requirements but are relevant:
- "skill": candidate skill name
- "relevant_to": which job skill(s) it could complement
- "reasoning": why this skill is transferable

### 3. interview_questions
4-8 targeted questions in TWO categories:

A) VERIFICATION (category: "verification"):
   - Skills claimed but with little evidence
   - Proficiency possibly inflated
B) TECHNICAL DEPTH (category: "technical_depth"):
   - Design/architecture questions on their strongest skills
   - "How would you..." scenarios combining multiple skills
   - Edge cases, trade-offs, failure scenarios

Each: "skill", "question", "reason", "category", "priority" (high/medium/low)

### 4. summary
- "strengths": array of 2-4 key strengths
- "risks": array of 1-3 concerns
- "overall_fit": 1-2 sentence assessment
- "recommendation": one of "strong_yes", "yes", "maybe", "no"

Return ONLY valid JSON.

    Expected output (4 sections):

    Section Contents
    skill_assessments Per-skill: match_level, evidence_summary, cv_quotes, confidence
    transferable_skills Skills not in JD but relevant, with reasoning
    interview_questions 4-8 questions (verification + technical_depth), with priority
    summary Strengths, risks, overall_fit text, recommendation

    Fallback: _fallback_analysis() builds assessments from deep_skills data using proficiency/justification mapping. No transferable skills. Generates verification questions for weak/unsubstantiated skills only. Returns recommendation: "maybe".

    6. Temporal Feature Extraction¶

    Source: app/services/temporal_extraction.py:12 — PROMPT

    Purpose: Extract years of experience from CV text.

    User message: CV text (truncated to 8,000 chars).

    Token limit: 256. Timeout: Default.

    Prompt text:

    Extract temporal/work-experience features from the CV text.
Return ONLY a JSON object with these keys (use null if not found):
"years_of_experience" (float, total professional experience),
"years_worked_abroad" (float, work outside home country),
"years_in_current_role" (float, if inferrable),
"years_in_management" (float, if inferrable).
No explanation, only valid JSON.

    Expected output:

    {"years_of_experience": 8.0, "years_worked_abroad": 2.0, "years_in_current_role": null, "years_in_management": 3.0}

    Fallback: Returns empty dict {} when LLM unavailable.

    7. OCR Text Extraction¶

    Source: app/services/ocr.py:21 — EXTRACT_TEXT_PROMPT

    Purpose: Extract text from document images using vision model.

    Model: Qwen2.5-VL-3B-Instruct (via vLLM, separate from the main LLM).

    Prompt text:

    Extract all text from this document in reading order. Preserve structure (headings, lists) where visible.

    Input: Base64-encoded image sent as image_url content type.

    Token limit: 4,096 per page. Timeout: 60s per page.

    Fallback: When OCR is unavailable, PDF parsing falls back to pdfplumber (text-based extraction only, no image support).

    Token limits and timeouts¶

    Prompt Input truncation Max output tokens Timeout
    CV Deep Extraction 32,000 chars 16,384 300s
    CV Simple Extraction 32,000 chars Default Default
    JD Skill Extraction 16,000 chars Default Default
    JD Generation N/A 2,048 120s
    Candidate Analysis 24,000 chars (CV) 16,384 300s
    Temporal Features 8,000 chars 256 Default
    OCR 1 image per call 4,096 60s/page

    How to modify prompts¶

    1. Edit the prompt variable in the source file
    2. No restart needed if using --reload (uvicorn watches for changes)
    3. Temperature is hardcoded at 0 — change in call_local_llm() if needed
    4. To add a new output field, update both the prompt text AND the corresponding _parse_* / _validate_* function

    Changing the model¶

    Set LLM_MODEL in .env. Any OpenAI-compatible model works. Larger models may improve extraction quality but increase latency. The prompts are designed for models in the 8B-70B range.

    9. Fallback Behavior Summary¶

    Component LLM Available LLM Unavailable
    CV Skill Extraction Deep extraction (proficiency + evidence) Phrase matching against skill library
    JD Skill Extraction LLM with decomposition + normalization Regex phrase matching, all marked required
    Candidate Analysis Full semantic analysis (4 sections) Binary presence mapping, basic questions
    Temporal Features LLM extracts years Empty dict (no temporal data)
    OCR Qwen2.5-VL vision model pdfplumber (text PDFs only)
