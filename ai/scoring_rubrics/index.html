    Scoring System Reference¶

    This document defines every scoring scale in VICA, how they combine, and how they flow into SPEA2 ranking and candidate reports.

    1. Proficiency Levels (CV Extraction)¶

    Assigned by the LLM during deep skill extraction (app/services/skill_extraction.py:33).

    Level Definition Typical CV evidence
    expert Multiple years of hands-on use, led projects, deep knowledge demonstrated "Led migration of...", "5+ years", "architected"
    proficient Clear practical experience, used in real projects "Developed features using...", "worked on X project"
    familiar Some exposure, coursework, or minor usage "Completed coursework in...", "assisted with..."
    mentioned Listed but no supporting evidence in the CV Appears in skills list only, no context

    Used in: Reports (skill detail cards). NOT used in SPEA2 ranking — SPEA2 uses binary presence only.

    2. Justification Levels (CV Extraction)¶

    Assigned by the LLM alongside proficiency (app/services/skill_extraction.py:44). Measures evidence quality independently from proficiency depth.

    Level Definition Example
    strong Multiple concrete examples, projects, or years of experience 3 projects mentioned, specific metrics cited
    moderate At least one concrete example or project One project or role described
    weak Mentioned in skills list or implied but limited evidence Listed in "Skills:" section only
    unsubstantiated Claimed but contradicted or no evidence at all Says "expert in X" with zero supporting detail

    Used in: Reports, fallback analysis logic, interview question generation (weak/unsubstantiated skills trigger verification questions).

    3. Match Levels (Candidate Analysis)¶

    Assigned by the LLM during post-SPEA2 semantic analysis (app/services/skill_matching.py:41). Assessed per job skill per candidate.

    Level Definition Report display
    strong_match Clear, direct evidence of this skill with real experience Green badge
    partial_match Some evidence but not deep enough, or related but not exact Yellow badge
    transferable Candidate has a closely related skill that could transfer Blue badge
    no_match No evidence of this or related skills Red badge

    Normalization (_normalize_match_level at line 191): The LLM may return variations — the code normalizes by: - Replacing spaces/hyphens with underscores - Mapping keywords: "strong" → strong_match, "partial" → partial_match, "transfer" → transferable - Default: no_match

    4. Recommendation Levels (Candidate Analysis)¶

    Assigned in the summary section of candidate analysis (app/services/skill_matching.py:87).

    Level Meaning
    strong_yes Excellent fit, proceed with interview — strong evidence across required skills
    yes Good fit, standard interview process
    maybe Mixed signals, needs clarification — some gaps or unsubstantiated claims
    no Not suitable — missing critical required skills

    Normalization (_normalize_recommendation at line 270): Maps LLM variations using keyword matching. Default: maybe.

    5. Confidence Levels¶

    Assigned per skill assessment in candidate analysis (app/services/skill_matching.py:54).

    Level Meaning
    high Clear, definitive assessment — strong evidence or clear absence
    medium Reasonably confident — some evidence but room for interpretation
    low Uncertain or ambiguous — limited CV data for this skill

    6. Interview Question Categories & Priority¶

    Generated in candidate analysis (app/services/skill_matching.py:63).

    Categories:

    Category Purpose When generated
    verification Probe skills claimed but with little evidence Weak/unsubstantiated justification, unclear proficiency
    technical_depth Test real depth on strongest skills Expert/proficient skills with strong evidence

    Priority levels:

    Priority Meaning
    high Critical to verify — unsubstantiated claims or core required skills
    medium Important to understand — moderate evidence gaps
    low Nice to explore — additional depth on strong skills

    7. How Scores Feed Into SPEA2¶

    Important: SPEA2 uses binary skill presence only. A candidate either has a skill (1) or doesn't (0). Proficiency, justification, match_level, and confidence are not used in ranking — they exist for the report only.

    graph LR
    CV["CV Text"] --> DE["Deep Extraction"]
    DE --> SN["Skill Names"]
    SN --> TN["Taxonomy<br/>Normalization"]
    TN --> BV["Binary Vector<br/>has / doesn't have"]
    BV --> SP["SPEA2 Ranking<br/>(binary × weights)"]

    DE -.->|"for reports only"| PROF["Proficiency"]
    DE -.->|"for reports only"| JUST["Justification"]
    DE -.->|"for reports only"| EVID["Evidence"]

    style SP fill:#fff3e0,stroke:#ef6c00

    The richer data (proficiency, evidence, justification) flows into: - Candidate reports: detailed skill cards with evidence citations - Post-match analysis: semantic match_level, transferable skills, interview questions - Fallback analysis: maps proficiency + justification to match_level when LLM unavailable

    8. SPEA2 Weights¶

    Defined in app/services/spea2.py:12.

    Base weights:

    Skill type Base weight
    Required WEIGHT_REQUIRED = 2.0
    Preferred WEIGHT_PREFERRED = 1.0

    Position factor: Skills earlier in the recruiter's ordered list get higher weight.

    position_factor = (n - position) / n    # position 0 = 1.0, last position ≈ 1/n
position_factor = max(position_factor, 0.1)   # minimum floor of 0.1

    Combined weight per skill:

    weight[j] = base_weight × position_factor

    Example with 5 skills (3 required, 2 preferred):

    Position Skill Required Base Pos factor Final weight
    0 Python Yes 2.0 1.0 2.000
    1 AWS Yes 2.0 0.8 1.600
    2 Docker Yes 2.0 0.6 1.200
    3 TypeScript No 1.0 0.4 0.400
    4 Redis No 1.0 0.2 0.200

    Objective per candidate:

    objectives[i, j] = weight[j] × binary(candidate i has skill j)

    SPEA2 then uses Pareto dominance over these weighted objectives to rank candidates. See SPEA2 Algorithm for the full theory.

    9. Fallback Scoring¶

    When LLM is unavailable for candidate analysis, _fallback_analysis() in app/services/skill_matching.py:298 maps existing deep_skills data to match levels:

    Proficiency Justification → Match level
    expert or proficient strong or moderate strong_match
    any other strong or moderate partial_match
    any weak or unsubstantiated partial_match
    (skill not found) — no_match

    Fallback also generates verification interview questions for every skill with weak or unsubstantiated justification (up to 6 questions). No transferable skills are detected. Summary recommendation defaults to "maybe" with overall_fit text: "Analysis performed without LLM — limited to binary skill matching."
